## Using Argos package for query development

Functions in the `argos` and tidyverse packages are documented:

- Argos package: You can access documentation for the `argos` functions by navigating to the lines of code where the function is defined (if you click on a function loaded in your Global Environment in RStudio or type View(function) into the console (e.g. View(compute_new)) this will open automatically)
- Tidyverse packages: You can access documentation for loaded packages in RStudio with the following syntax: ?function, e.g. ?ggplot. There is also extensive documentation available online, for example: Tidyverse website
- [RStudio Cheatsheets for various packages](https://rstudio.com/resources/cheatsheets/)

## **Developing your own functions:**

* Develop functions that manipulate data expressions
* One task per function
* Use parameters to allow subsetting and debugging
* Few restrictions on scope or approach
* Where should the function be located?
* * For simpler queries, all functions can be located in the cohorts.R script
* * For more complicated queries, functions can be organized into several transparently-named scripts which start with "cohort_". The contents of any files which start with "cohort_" will be automatically sourced when the request is executed.
* How to document?
* * [roxygen2](http://r-pkgs.had.co.nz/man.html) style documentation
* Separate out functions which prep code and those which analyze code
* If site mention is required in code, mask early so the code can be shared

## **Optimizing code:**

* Materialize intermediates and use indices as needed
* Beware poorly written loops
* Consider where the processing is taking place and where it should be taking place (e.g. locally after a collect() or on the database)

## **Reviewing code (including your own):**

Questions to ask:

* Does it do what it is supposed to do?
* Is it well-written (succinct, interpretable names)?
* Is it documented?
* Are there redundancies or places where chunks of code can be consolidated?
* Are there "version specific" limitations?
* Is is reproducible? (Do you need to set the seed for any randomization?)

## **Gotchas**

* Watch out for places where the same R code will produce different results locally versus on the database:
* Operators used in regex, e.g. str_detect(variables, "this|that")
* Any function that relies on ordering that is not specified, e.g. row_number() in a window function, e.g. group_by(person_id) %>% arrange(date) %>% mutate(row_num = row_number()) where more than one row per date in some cases
* Careful when collecting and writing to the database that the fields are of the correct type
* Beware "timestamp without time zone" and "timestamp with time zone" - matching across these different types won't work properly
* Careful using between(x, value1, value2) where the values aren't constants but vary by row.  It looks like the database recognizes per-row comparisons but local R code will respond to global true/false conditions unless the data frame/tibble is grouped.  

## **Other helpful hints**

* How to use SQL in your R code
* * Example manipulating datetimes within the standard package
    * mutate(adt_datetime_12_hours = sql("(adt_datetime + interval '12 hour')"))
* Including indices when outputting tables to the db can speed up subsequent joins and other operations.
* * output_tbl('my_table', indices=list(c('person_id', 'other_variable')))
* When working with tables in a specific schema, be sure to specify the schema and results name tag in run.R
* * line 120 config('results_schema', ' *study schema* ')
* * lines 135-136 config('results_name_tag', ' *your name tag* ')
